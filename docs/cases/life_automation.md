所属类别：Life Automation

  
合成方法
  * 场景分析与最小环境集确定：日程与计划是对自动化执行需求较大的日常生活场景之一，因此适合构造用于OpenClaw能力评测的环境。具体场景构建中，我们选择了飞机出行这种广泛使用的交通工具，并添加了用户安排日程常用的Calendar管理应用和用于沟通交流的邮件应用。结合这三种日常应用，可以衍生出多种不同难度、执行长度的任务。
  * 数据合成流程：
    1. 环境合成：分别开发三种Mock应用，参照真实应用进行应用设计，并采用标准的B/S架构进行开发，前端采用React.js框架，后端采用Flask框架，数据库使用SQLite3，保证了Mock环境接近仿真的同时，保持轻量的部署。
    2. Case合成：本场景下，不同的Case可以直接复用环境合成中得到的Mock应用的各种组合。因此Case合成只需要完成：任务元信息定义、任务提示词编写、环境安装脚本开发（包括OpenClaw所需依赖和Mock环境的安装和数据注入）、测试脚本开发


思考/Highlight：核心是Sim2Real：相比于之前的评测，我们关注<u>测试的真实性和评测的可验证性</u>，从而更精准的衡量基座模型驾驭OpenClaw的能力
    1.测试的真实性：
      a. 模拟环境的真实性：相比于之前的工作（PinchBench/Claw-Eval），我们开发的模拟环境包括高度仿真的前端和后端，是一个更完整、更符合真实场景的服务，因此更接近OpenClaw在实际环境中的能力。
      b. 场景的真实性：我们设计的场景均来自于日常用户的高频率场景，并且是对OpenClaw这种自动化执行有需求的场景，从而保证评测有意义，且与用户日常使用体验对齐。
      c. 合成任务的真实性：我们合成的每个任务中都有大量预先注入的数据，从而保障环境分布更接近真实应用。此外，我们合成的任务覆盖不同的难度和执行链长度，可评价模型应对不同层次任务的表现。
      d. 数据时效性解耦：由于Life Automation这个场景中涉及到的任务往往与时间相关，而之前的评测往往用静态数据并且尽可能回避时间相关的任务，而这个可能导致评测与真实场景存在偏差。为了解决这个问题，我们在评测运行时的mock环境安装时再往数据库中注入预定义数据，并且注入数据的脚本中的时间字段均基于时间偏移量计算，从而使得在任何时间运行评测，mock应用中信息的时间都是合理的且是与任务统一的。因此，我们的评测中会有与时间相关的任务。
    2.最小环境集确定 + 环境复用
      a. 与之前的评测不同，我们在分析日程与计划场景时，首先确定了可开展真实任务的最小环境集（Email + Airline + Calendar），保证了任务的跨环境需求，从而避免出现评测场景与真实场景存在明显差距的问题。
      b. 我们的设计准则要求：不同的case使用的各个环境是相同的，只有数据库文件不同（.db文件），这样可以避免冗余开发，提高任务合成效率。
    3. 可验证性：不同的Case在评测时，可以认为环境发生变化的只有数据库文件，因此我们在检验任务是否完成时，只需要检查数据库文件中是否可以观测到任务完成所期望的数据，从而避免发生奖励hacking。


* Case 1
  * 名称：email-writing
  * 描述：让openclaw帮助用户给指定收件人编辑并发送邮件
  * 涉及环境：Email
  * 难度：Easy（单环境+少步数）
* Case 2
  * 名称：email-reply
  * 描述：让openclaw检查是否存在期望邮件，如果存在则编辑并发送回信
  * 涉及环境：Email
  * 难度：Easy（单环境+少步数）
* Case 3
  * 名称：flight-booking
  * 描述：让openclaw帮助用户预定指定日期和起始点的机票
  * 涉及环境：Airline
  * 难度：Easy（单环境+少步数）
* Case 4
  * 名称：flight-seat-selection
  * 描述：让openclaw帮助用户完成指定预定航班的值机选座，需要首先从邮件中找到预定航班的信息，并在航司网站按照用户要求进行选座
  * 涉及环境：Email + Airline
  * 难度：Medium（多环境+少步数）
* Case 5
  * 名称：flight-seat-selection-failed
  * 描述：让openclaw帮助用户完成指定预定航班的值机选座，需要首先从邮件中找到预定航班的信息，并在航司网站按照用户要求进行选座。但是用户的选座要求无法被满足，期望openclaw可以找到最佳替代方案
  * 涉及环境：Email + Airline
  * 难度：Hard（多环境+多步数）
* Case 6
  * 名称：flight-cancel-claim
  * 描述：让openclaw检查预定航班的状态是否正常。如果出现被取消的情况，需要openclaw帮助用户按照航司网站上的公告申请理赔
  * 涉及环境：Email + Airline
  * 难度：Hard（多环境+多步数）
* Case 7
  * 名称：flight-info-change-notice
  * 描述：让openclaw检查邮箱，是否有关于航司发来的预定航班状态变化的通知，如有，则需要检查日程应用中是否存在受到影响的计划，并能够提醒同程伙伴
  * 涉及环境：Email + Airline + Calendar
  * 难度：Hard（多环境+多步数）
* Case 8
  * 名称：baggage-tracking-application
  * 描述：让openclaw在航司网站上找寻方法，可以向航司登记自己丢失的行李
  * 涉及环境：Airline
  * 难度：Medium（单环境+多步数）
* Case 9
  * 名称：schedule-change-request
  * 描述：用户因事导致日程发生变化，因此需要openclaw帮助用户找出Calendar上存在冲突的日程，并向他们写邮件告知日程变化并解释原因
  * 涉及环境：Email + Calendar
  * 难度：Hard（多环境+多步数）
